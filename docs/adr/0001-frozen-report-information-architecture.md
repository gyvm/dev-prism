# ADR 0001: 凍結レポートの情報設計（問い別 IA への再編）

- ステータス: Implemented（2026-06-22, branch `claude/frozen-report-v2`）
- 日付: 2026-06-22
- 決定方法: grilling（壁打ち）で確定
- 関連: [[design-modernization-plan]]、`src/pipeline/stages/render.tsx`、`src/analyses/dora-metrics/`

## コンテキスト

凍結レポート（Dev Prism 週次振り返り）の内容に、利用者から3つの不満があった。

1. **重複** — 同じリードタイムが DORA カードと Dev Prism サマリーで二重に出る。
2. **順番** — 確認する項目の並びが整理されていない。
3. **種類別表示** — 項目を種類ごとに分けて見たい。

現状レポートはセクションを **描画タイプ**（数字カード → AI markdown → チャート）で縦割りしていた。このため「リードタイムの数字」と「その数字に効いた PR の話」が別の階に分かれ、原因と結果が離れていた。

### 利用文脈（grilling で確定した前提）

- 一次利用は **チームの週次振り返り MTG**（10人以下、テックリードが進行、画面共有）。
- チームはリードタイム・デプロイの **目標を立て、毎週 達成/未達を振り返って上司に報告**している。
- 上司報告は **レポートの数字部分をコピーして別ドキュメントへ**貼る運用。→ 報告専用ビューは不要、先頭の数字をコピーしやすくすれば足りる。
- 体験の流れは **「数字（体温）→ 理由 → 会話」**。

### 計測上の事実（実装を読んで確認）

`src/analyses/dora-metrics/internal/dora.ts` の DORA 4指標は PR 代理値で、3つが不正直だった。

- `deploymentFrequency` = 今週のマージ数（デプロイではない）。
- `leadTimeForChangesHours` = PR 中央値リードタイム（本物。サマリーと重複）。
- `changeFailureRatePercent` / `mttrHours` = `hotfix`/`revert`/`incident` ラベル依存。運用が無いチームでは大体 0% か空。

標準ラベル（DORA）を非標準データ（PR 代理値）に貼ると、DORA の権威を借りて誤読を生む（「デプロイ頻度」を名乗るマージ数、「変更失敗率 0%」に見える未計測）。

## 決定

### 1. 描画タイプ縦割りをやめ、**問い別（数字 → 理由 → 会話）** の3セクションに再編する

#### ① 開発メトリクス（数字＝報告コピー元 / 振り返りの結論）

- **4指標**（`dora-metrics` を正直化）— 各 *数値 + 前週比*
  - デプロイ頻度 ＝ **デフォルトブランチへのマージ数**（正直なラベルに改名）
  - 変更リードタイム（中央値）
  - 変更失敗率 ＝ **`Revert "…"` タイトル自動検知**ベース（ラベル依存を廃止）
  - MTTR ＝ 同 Revert 検知ベース
- **PR タイムライン**（`pr-timeline` ガント）
- **その数字に効いた PR**（AI `00_flow-analyst`）— *今週マージ/クローズされた*、リードタイムの長い PR。内訳はレビュー依頼までの待ち等の**フェーズ分解**。
- **この結果を受けての問い**（AI）— 効率化・目標見直しなど **数値目標に対するアクション**。

#### ② 開発内容の要約（何が起きたか＝振り返りの素材）

- **全体進捗**（AI `01_project-progress`）— ざっくりグループ化。
- **フォローアップが必要な PR / 拾っておきたい動き**（AI `02_follow-up-prs` ＋ `03_debated-prs` を統合）— 必要なアクション、小さく価値ある PR、**議論が多い PR**。
- **来週確認したいこと**（AI/code）— *まだ開いている*：長く放置の PR / 未解決レビュー / コメント後に停止。**振り返りの気づきに対するアクション**。

#### ③ PR レビュー（レビューの偏り）

- **レビュー相関図**（`review-correlation` bipartite）。
- **レビューバランスの AI コメント**。

### 2. 重複の解消（不満への直接の回答）

- **リードタイム二重表示** → `dora-metrics` と `dev-prism-summary` を ①の数字ブロック1つに統合。
- **「遅い PR」と「放置 PR」の重複** → **closed（①その数字に効いた）/ open（②来週確認）** で分離。これが両者の本質的な分割線。
- **議論 PR** は速度軸から外し ②フォローアップへ畳む（`03_debated-prs` 単独セクションを廃止）。

### 3. アクションは意図的に2本（重複ではない）

- **数値目標に対するアクション**（①「この結果を受けての問い」）。
- **振り返りの気づきに対するアクション**（②「来週確認したいこと」）。
- ジョブが異なるため統合しない。

### 4. DORA の正直化

- 偽ラベルを捨て正直なラベルへ。変更失敗率/MTTR は **`Revert "…"` タイトル自動検知**で初期実装（ラベル運用ゼロでも成立）。今後ブラッシュアップ。

## 今回スコープ外（後回し）

- **目標値の入力 / 達成・未達判定** — 数値＋前週比のみで成立。目標値は将来 `repositories` と同じ場所（GitHub Action の `with:` / args）に置く（args ファースト方針と整合、[[product-direction-config]]）。
- **PR サイズによる遅さ評価** — 大きいが速い PR もあり、リードタイムの主因とは限らない。
- **自動生成コードの除外** — サイズ評価を外したため不要に。

## 影響

- `src/pipeline/stages/render.tsx` の `renderReportHtml` を、`AI_SECTION_ORDER` / `DETAIL_SECTION_ORDER` ベースの描画タイプ縦割りから、3セクション（問い別）構成へ再編。
- `src/analyses/dora-metrics/internal/dora.ts`（in-memory）と `src/analyses/dora-metrics/query.ts`（SQL）の**両方**を Revert タイトル検知（`title LIKE 'Revert "%'`）に変更、ラベル定数 `FAILURE_LABELS` を両経路で廃止（parity 契約のため必須）。指標ラベルを正直化（metric-cards）。
- `dev-prism-summary` と `dora-metrics` のリードタイム表示を統合（重複削除）。
- AI プロンプト `03_debated-prs` を `02_follow-up-prs` 文脈へ統合。

## 実装メモ（2026-06-22）

- ①〜③の3バンドは `render.tsx` の `renderBand` 配列で実装（順序の単一の源）。
- **debated は独立セクションを廃止し、`follow-up-prs` プロンプトの `💬 議論` カテゴリへ統合**（ユーザー Q15 の意図どおり）。決定的な「議論が多かったPR」候補リストは `dev-prism-summary` に残置。
- **`review-balance`（AI）を新設**し ③PRレビューに配置。`flow-analyst` のセクション見出しは「その数字に効いたPR」。
- **PR サイズ（`largePrs`）を全廃**（compute/query/types/renderer/test）。
- 残課題（軽微・別PR）: `metric-cards` 内の「DORAメトリクス」h2 がバンド h2 と併存（Explore と共有のため据え置き）。プロンプト本文の先頭 H2 は render が除去して扱う（本文側の整形は未実施）。

## 選択肢と却下理由

- **DORA をそのまま維持** — 偽の3指標が誤読を生むため却下。
- **本物の DORA（Deployments/Incidents API を追加取得）** — v0.1.0 では過剰。Revert 検知で代替し将来検討。
- **描画タイプ縦割りのまま順番だけ調整** — 原因（理由）と結果（数字）が離れたままで「重複感」が消えないため却下。
